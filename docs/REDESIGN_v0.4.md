# invest_notify 再設計案 v0.4 — Radar を「ランキング通知」から「変化（イベント）通知」へ

> ステータス: **提案（要レビュー・要合意）**。v0.3 の B 系統（Multibagger Radar）を 3 週間運用した結果を受けての改修案。
> 対象: `docs/REDESIGN_v0.3.md` で実装した B 系統の週次メール / スコアリング / 状態遷移。
> ゴール: **「前回通知以降に動きのあったものだけを、人間が見切れる件数で通知する」**。

---

## 0. 何が問題なのか（運用 3 週間の実データ）

3 回（2026-05-12 / 05-19 / 05-26）のメールと `candidates.json` / `_state.json` を分析した。
ユーザーの 2 つの不満は、いずれも**実データで裏が取れた構造的欠陥**である。

### 不満①「毎回同じ会社が来る」 → 事実。むしろ後半 2 週は上位 10 件が完全一致

「今週のトリガ」セクションは `triggers[:10]`（total 降順の上位 10）を出している。実際の上位 10:

| 週 | 上位 10（トリガ） |
|----|----|
| 05-12 | SPHR, INSW, TALK, HCC, LPG, CDNA, MRNA, FLYW, FDMT, SKYT |
| 05-19 | SPHR, INSW, STAA, SKYT, HCC, LPG, GNK, AMN, CDNA, XMTR |
| 05-26 | SPHR, INSW, STAA, SKYT, HCC, LPG, GNK, CDNA, AMN, XMTR |

- **05-19 と 05-26 の上位 10 は順序違いだけで 10/10 一致。**
- 3 週すべてに出続けた銘柄: CDNA, HCC, INSW, LPG, SKYT, SPHR（6 件）。

### 不満②「件数が多すぎる（トリガ548 / 候補1259 / 昇格57 / 過熱138）」 → 事実。閾値とスコア設計の飽和が原因

| 週 | トリガ | 候補 | 昇格 | 過熱 |
|----|----|----|----|----|
| 05-12 | 548 | 1279 | 52 | 131 |
| 05-19 | 524 | 1277 | 109 | 134 |
| 05-26 | 548 | 1259 | 57 | 138 |

ユニバース 2135 銘柄のうち **約 25% が trigger** という状態。

### 根本原因 A: スコアがほぼ静的（だから上位が固定される）

週をまたいだ `total` の変化量 `|Δtotal|`（05-19 → 05-26、2135 銘柄）:

- 中央値 **0.0000** / p90 **0.0222** / 最大 0.2064
- `|Δtotal| ≥ 0.05` だったのは **67/2135 銘柄（3%）** のみ。

つまり **97% の銘柄はスコアが週次でほぼ動かない**。ファンダ（売上・利益率・希薄化）は決算時しか更新されず、唯一週次で動く momentum も 9 分の 1 の寄与しかない。**上位ランキングが固定になるのは数学的必然**であり、上位 10 を出し続ける限り「毎回同じ会社」は避けられない。

### 根本原因 B: 9 シグナル中 4 つしか効いていない（だから 548 件も通る）

ユニバース全体での各シグナルの分布（05-26）:

| シグナル | 平均 | 標準偏差 | 1.0 に張り付く割合 | 評価 |
|----|----|----|----|----|
| `size_band` | 1.00 | **0.000** | 100% | **完全に無情報**（ユニバース通過条件そのもの） |
| `skepticism` | 0.98 | **0.055** | 77% | ほぼ定数 |
| `valuation_room` | 0.90 | 0.270 | 84% | 飽和ぎみ |
| `theme_link` | 0.81 | 0.243 | 62% | 飽和ぎみ |
| `non_dilutive` | 0.81 | 0.361 | 71% | 飽和ぎみ |
| `growth` | 0.45 | 0.310 | 15% | 有効 |
| `margin_improvement` | 0.50 | 0.318 | 15% | 有効 |
| `consistency_4q` | 0.49 | 0.289 | 11% | 有効 |
| `momentum` | 0.46 | 0.276 | 13% | 有効 |

9 つを単純平均しているため、**5 つの飽和シグナルが常時 0.8〜1.0 の下駄を履かせる**。その結果、growth/margin が平凡でも total が 0.75 を超えてしまい、ユニバースの 1/4 が trigger になる。trigger の total 分布は min 0.7504 / 中央値 0.798 と、**閾値 0.75 のすぐ上にびっしり詰まっている**。

### 根本原因 C: 「昇格」は本物の動きではなく境界フラッピング

05-12 → 05-19 の `candidate → trigger` 昇格 72 件のうち、**37 件（51%）が total 0.78 未満**（0.75 の床から 0.03 以内）。
trigger 集合自体は週次で 506/524 が据え置き（追加 42 / 脱落 18）。
→ 「昇格 N 件」の大半は、**意味の薄い 0.75 ラインを決算更新のわずかな揺れでまたいだだけ**で、10 バガー候補としての“動き”ではない。

### まとめ

現状のメールは **「ほぼ静的なスコアの上位ランキングの再掲」** であって、**「変化の通知」になっていない**。ユーザーが欲しいのは後者。設計思想を「レベル（水準）通知」から「イベント（変化）通知」へ転換する必要がある。

## 1. あるべき姿（提案の核）

### 方針: 「レベル通知」→「イベント通知」

メールの主役を **「スコア上位ランキング」から「前回通知以降に新しく発生した離散イベント（最大 N 件）」** に差し替える。
静的な高スコア銘柄は「殿堂入り」として 1 行に畳み、毎週は鳴らさない。

3 つの設計原則:

1. **イベント駆動** — 「水準が高い」ではなく「前回から変わった」を通知する。
2. **新規性の抑制（クールダウン）** — 一度通知した銘柄は、新しい根拠（新決算 or 上位ティア昇格）が出るまで再掲しない。
3. **人間スケールの件数** — メール本文は最大 10 イベント。548/1259 のような全体カウントはフッターに格下げ。

---

## 2. イベントの定義（“動き”の源泉に直結させる）

“動き”は本質的に次の 3 つからしか生まれない。これを離散イベントとして明示的に検出する。

| イベント種別 | 検出条件（前回 state と比較） | 直感 |
|----|----|----|
| `EARNINGS_ACCEL` | **新しい決算期を検出**（保存した fiscal date が更新）かつ growth / margin / consistency のいずれかが改善 | 「数字が決算ごとに強くなっている」＝ C7。最重要 |
| `BREAKOUT` | 200 日線を**新規に**奪還 / 底からの倍率が **2〜5x ゾーンに新規進入** / 出来高比が急増 — 前週すでにその状態だったものは除外 | 胴体後半の点火 |
| `TIER_UP` | ヒステリシス付きでティア境界を超え、かつ「初回」または「クールダウン明け」 | 観察→候補→トリガの実質的な格上げ |
| `OVERHEAT` / `FADE`（任意） | 注視・通知済み銘柄が過熱入り / 失速 | 手仕舞い・降格の注意喚起 |

ポイント:
- **既にその状態だった銘柄はイベントを生まない**。SPHR が 3 週連続 trigger でも、新決算もブレイクもなければ「イベントなし」＝メールに出ない。これが不満①の直接の解決。
- 各イベントは「いつ・何が・どれだけ変わったか」を必ず持つ（例: 売上 YoY +37→+77%、200 日線を 5/20 に奪還、底から 2.0→2.3x）。

---

## 3. 新規性の抑制（クールダウン）

state ファイルを拡張し、銘柄ごとに次を保存する:

```jsonc
{
  "TICKER": {
    "state": "trigger",
    "total": 0.83,
    "last_fiscal_date": "2026-03-31",   // 決算デルタ検出用（新規）
    "last_notified_at": "2026-05-12",   // クールダウン判定用（新規）
    "last_notified_event": "TIER_UP",    // 何で通知したか（新規）
    "momentum_flags": {"over_sma200": true, "in_2_5x": true}  // ブレイク新規性判定用（新規）
  }
}
```

ルール:

- ある銘柄をヘッドラインに出したら `last_notified_at` を記録する。
- 以降 **N 週間（既定 8 週＝四半期相当）はヘッドラインに再掲しない**。ただし例外として、
  - **新しい決算が出た**（`last_fiscal_date` が変わった）→ クールダウンを無視して再評価・再通知できる。
  - **より上位のティアに進んだ**（candidate → trigger など実質的な昇格）→ 再通知できる。
- これにより「同じ顔ぶれの反復」を根絶しつつ、本当に新しい材料が出たときだけ再浮上させる。

## 4. スコア設計の修正（飽和の解消・ゲート化）

不満②（件数過多）の根治。現行の「9 シグナル単純平均」をやめ、**ゲート + ランキング**の 2 段構成にする。

### 4.1 無情報シグナルを得点から外す

- `size_band` は **得点から除外**し、ユニバース通過条件（ゲート）に降格。std=0 で点数化の意味がない。
- `skepticism` も std=0.055 とほぼ定数。重みを下げるか、補助情報（メール表示のみ）に降格。

### 4.2 必須ゲート方式（飽和シグナルの下駄を物理的に潰す）

`trigger` は平均点で決めず、**コア 4 シグナルの最低ライン（AND 条件）を必須**にする。例:

```
trigger になるには（全て必須）:
  growth          >= 0.6     (直近 YoY +18% 以上)
  consistency_4q  >= 0.67    (4Q 中 3Q 以上で加速)
  momentum        >= 0.7     (200日線奪還 + 底から 1.5x 以上)
AND 除外:
  valuation_room  >  0       (PSR が過熱でない)
  non_dilutive    >  0       (希薄化が壊れていない)
```

ゲートを通った銘柄のみ、`growth / margin / consistency / momentum` の重み付き合算で**順位付け**する（飽和シグナルは順位にもスコアにも使わない）。これで「飽和で下駄を履いて通過」が消え、trigger は数百件 → 数十件規模に落ちる見込み。

### 4.3 閾値の引き上げ（v0.3 §9 の宿題）

v0.3 §9 が既に「trigger を total≥0.85 / momentum≥0.7 に上げる」「growth/margin に強い重みを」と示唆していた。本提案はそれをゲート方式で具体化したもの。

## 5. 週次メールの新フォーマット（案）

```
件名: [Radar Weekly] 2026-05-26 新着7（決算3 / ブレイク2 / 昇格2）

== 今週の新着イベント（7 件） ==

1) [決算加速] NTRA  Natera  $XXB / Healthcare
   - 新決算 2026-03-31 を検出
   - 売上 YoY: +25% → +41%（加速）/ 営業利益率: +3 → +9pp
   - 株価: 底から 2.3x / 200日線 +12% / 出来高比 1.4x
   - 前回通知: なし（初出）
   - 次の確認: 次回決算（推定 2026-07）

2) [ブレイク] INDI  indie Semiconductor  $1.2B / Technology
   - 200 日線を 5/22 に新規奪還 / 底から 1.8x → 2.4x（2〜5x ゾーン入り）
   - 売上 YoY: +30%（前回から変化なし）
   ...

3) [昇格] ABUS  candidate → trigger（ゲート全通過: growth0.8 / consistency0.75 / momentum0.7）
   ...

== 殿堂入り（高スコア継続・新規材料なし、参考） ==
SPHR, INSW, STAA, SKYT, HCC, LPG  ← 詳細は前回までを参照。新決算が出たら再掲します。

== 状態遷移サマリ ==
新規昇格 2 / 降格 1 / 過熱入り 3

--
（参考）ユニバース 2135 / trigger 41 / candidate 180 / age 19日
```

変更点:
- **主役 = 新着イベント**。各イベントに「何が・いつ・どれだけ変わったか」と「前回通知の有無」を明記。
- **「殿堂入り」**で静的高スコア銘柄を 1 行に圧縮（情報は捨てない、毎週は鳴らさない）。
- 件名は **新着イベント数 + 内訳**。全体カウント（548/1259…）はフッターに格下げ。
- 「昇格」はゲート全通過のものだけ、境界フラッピングは出さない。

## 8. 要決定事項（ユーザー判断が必要なパラメータ）

実装前に以下を決めたい（叩き台の既定値つき）:

1. **メール 1 通あたりの最大イベント数**: 既定 **10**。
2. **クールダウン期間**: 既定 **8 週**（≒1 四半期。新決算 or 上位ティア昇格で解除）。
3. **trigger ゲートのライン**: growth≥0.6 / consistency≥0.67 / momentum≥0.7（§4.2）。緩い/厳しいの調整。
4. **「殿堂入り」を残すか**: 完全に隠す案も可。既定は 1 行で残す。
5. **過熱/失速（FADE）通知を入れるか**: 保有していない前提なら不要かも。

## 9. 付録: 本提案が 3 週間データに与える効果（想定）

- 上位 10 固定化 → **解消**（イベントがなければ鳴らない。05-19/05-26 のように材料がなければ「新着 0〜数件」になる）。
- 548 trigger → ゲート化で**数十件**に圧縮、メール掲載はうち新着イベントのみ。
- 「昇格 52〜109」→ ゲート全通過の実質昇格のみ（**1 桁〜十数件**）、境界フラッピング除去。
- ユーザーの読む量: 毎週「10 件の固定ランキング + 50〜100 行の昇格羅列」→「数件の新着イベント + 1 行の殿堂入り」。
