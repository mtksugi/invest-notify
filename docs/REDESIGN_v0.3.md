# invest_notify 再設計案 v0.3 — 「10バガー候補通知」への原点回帰

> ステータス: 提案（実装前）
> 対象: 同梱の `SPEC_MVP_v0.2.md` を置き換える次期仕様
> ゴール: README 冒頭にある本来の目的 — **「まだ織り込まれていない可能性のある事象」を抽出して、自分が判断できる形に圧縮して通知する** — を、明示的に **「10バガー候補（テーマ初期〜中盤の中小型株）の発見」** に振り直す。

---

## 0. 何が問題なのか（現状と目的のズレ）

PR #4, #5 でノイズ削減と確度評価は進んだが、いま毎日届くメールの実態は次のとおり。

- 当日の地政学/政策ヘッドライン
- 当日の 8-K（IR）見出し
- 当日の訴訟/当局発表
- 注視ティッカーの当日ニュース

これは **「日次ニュースモニタ」** であり、**「次の PLTR / APP / SMCI / VRT / CVNA / RKLB / CLS を見つける装置」** ではない。

ユーザーが本当に欲しいのは（要約）：

> **時価総額 10〜100 億ドル程度の、巨大テーマに接続した、まだ市場に半信半疑で見られている会社**
> を、株価が 2〜5 倍になりつつある（あるいは数字が加速し始めた）**胴体の後半** で拾う。

これに対し、現アプリは

- 「**今日のニュース**」を入力にしている → テーマ初期化のシグナルは元々入ってこない
- 「**当日 1 回**」のスナップショット判定 → "売上加速 3 四半期連続" のような **時系列の連続性** が見えない
- 評価軸が **当日の post-return (5〜10営業日)** → "数か月〜2年" の中盤モメンタム評価ができない
- 入力が **RSS 100〜200 件** → 地政学・マクロが大半で、テーマ周辺の中小型株が拾えない

つまり、構造的に「日次ニュースの中から 10 バガー候補が偶然降ってくるのを待つ」装置になっており、毎日の通知に該当銘柄が出てこないのは当然。

---

## 1. 仕切り直しの基本方針

「日次ニュース」と「10 バガー候補」は **別物** として扱う。
ひとつのパイプラインに混ぜず、**2 系統に分離する**。

| 系統                      | 目的                                                 | 周期      | 入口                                                         | 出口                              |
| ----------------------- | -------------------------------------------------- | ------- | ---------------------------------------------------------- | ------------------------------- |
| A. **Daily Watch（既存改修）** | 場況の把握 / 注視銘柄の動き / 当日の確度高イベント                       | 毎日      | RSS（ニュース/IR/訴訟/SNS）                                        | 既存メール（ただし「主目的ではない」位置づけに格下げ）     |
| B. **Multibagger Radar（新規・主役）** | **テーマ初期〜中盤の中小型株を継続的に発掘**して、芽が出た時だけ通知 | 週次 + 四半期 | ユニバース（時価総額バンド付き米株/日本株） + 業績/決算/価格モメンタム + テーマ辞書 + 一次ソース | 候補銘柄の watchlist 更新 + "芽の出た銘柄"の通知 |

つまり、**毎日の通知に 10 バガー候補が出ない** のは仕様であって異常ではない。
**B の仕組みは「毎日鳴る装置」ではなく、「条件が揃った時にだけ鳴る装置」** にすべき。
鳴らない日が普通であり、鳴った時の濃度が高い、というのが正しい姿。

## 2. Multibagger Radar（B 系統）の設計

### 2.1 ユーザーが要求した「狙うべき銘柄」の条件（再整理）

ユーザーの引用から、機械的に判定可能な条件に落としたもの:

| #  | 条件                              | 取得難度  | 機械判定 |
| -- | ------------------------------- | ----- | ---- |
| C1 | 時価総額バンド（5億〜200億ドル）              | 易     | ◎    |
| C2 | 巨大テーマに接続している                    | 中     | △（テーマ辞書 + 文書類似度） |
| C3 | 売上成長がすでに数字に出ている（YoY/QoQ）        | 中     | ◎（決算 API） |
| C4 | 粗利率/営業利益率が改善している（営業レバレッジ）        | 中     | ◎    |
| C5 | バリュエーション過熱（PSR/PER）になっていない      | 中     | ◎    |
| C6 | 増資で株主価値を壊していない（希薄化）             | 中     | ◎（発行済株式の推移） |
| C7 | 直近 3〜4 四半期で「仮説が決算ごとに強くなっている」    | 高     | ○（時系列スコア） |
| C8 | 株価モメンタム：底から 2〜5 倍 / 200日線越え     | 易     | ◎    |
| C9 | 市場がまだ「地味」「怪しい」「一時的」と見ている         | 高     | △（カバレッジ件数 + ショート比率 + 出来高比） |

「悪い中盤」の条件（除外側）:

- 株価だけ先に 5〜10 倍、売上成長は弱い
- PSR/PER が異常
- SNS で騒がれまくっている
- 直近決算で大きく外した

### 2.3 メール出力のフォーマット（B 系統）

毎日の A メールとは **別件名・別フォーマット** にする。

```
件名: [Radar] YYYY-MM-DD 候補N件 / 芽トリガM件

[芽トリガ] AAAA  時価総額 $XX億 / テーマ: 〇〇
- 直近4Q 売上 YoY: +18 → +27 → +34 → +41%（加速）
- 営業利益率: -3 → +1 → +5 → +9%（黒字転換）
- 株価: 底から 3.2x、200日線 +12%、出来高 直近20日比 +180%
- ガイダンス: 通期売上を +12% 上方修正（XX/YY 開示）
- 半信半疑の根拠: アナリストカバレッジ 4社のみ / ショート比率 12%
- リスク: PSR 11x（やや過熱の入口）/ 主要顧客集中度 38%
- 一次ソース: <URL>
- 次の確認: 次回決算（YYYY-MM-DD）でガイダンス維持できるか
```

判断材料として人間が読める粒度で、**「数か月後の値動き」** に効く要素だけ並べる。

### 2.4 一過性ノイズの抑制

- 半年以上、業績 / モメンタム / テーマ接続度のいずれかが連続して改善している銘柄だけを「芽トリガ」に上げる
- 単発の急騰（2 営業日で +30% など）は除外
- 「胴体の後半を取りに行く」前提なので、底値（直近 3 か月で 0% モメンタム）はあえて拾わない

## 3. Daily Watch（A 系統）の位置づけの更新

A は廃止しない。ただし役割を再定義する。

- **主役は B**。A は「毎日の場況把握 + 注視銘柄の動向」モニタとして残す
- A の通知件数上限は据え置き（confirmed 0〜3 / early_warning 0〜3）
- A の中で **B のユニバースと突き合わせ** て、A のニュースが B 候補銘柄に当たったら自動でフラグを付ける（"radar hit"）
  - これにより「日次ニュースから候補銘柄を増やす」フィードバックループが生まれる
- 件名/構成を、**「今日の値動き材料」と「数か月先の構造変化」を視覚的に分ける** ように改修
- README/SPEC の冒頭で「主目的は B、A は補助」と明記する

## 5. 実装上の注意（重要なリスク）

- **無料の決算 API は精度・整合性に難** → 主要候補だけ手動メンテのフォールバックを持つ
- **時価総額バンドは USD/JPY のレートで歪む** → 日本株は円ベースで別バンド
- **テーマ辞書は半年ごとに腐る** → ユーザーが手で編集できる YAML にする
- **「半信半疑」スコアは精度が出にくい** → MVP では「カバレッジ件数 + 出来高比」だけで近似してよい
- **B-5 の連発防止** → 同一銘柄の通知間隔を最低 30 日空ける
- **A と B の通知は同じメールに混ぜない**（軸の違う情報を視覚的に分けないと、また「ノイズが多い」になる）

## 6. 段階導入（実装順）

各段階で「動く範囲を増やす」ことを優先する。
（カレンダー時間ではなく、**変更の侵襲度** で並べている）

1. **Phase 0（このPR）**: 本ドキュメント `docs/REDESIGN_v0.3.md` を追加し、READMEの目的説明を「主目的=Multibagger Radar、補助=Daily Watch」に更新する。コードは未変更。
2. **Phase 1（最小実装）**:
   - `invest_notify radar build-universe` サブコマンド（時価総額バンドのユニバース生成）
   - `invest_notify radar fetch-fundamentals` サブコマンド（候補のみのファンダ取得）
   - 出力は JSON のみ。通知は出さない。
3. **Phase 2（スコアリング）**:
   - C1〜C9 のスコア計算（最初は C1, C3, C4, C8 だけで構わない）
   - `data/radar/candidates.json` を生成
4. **Phase 3（芽トリガ + メール）**:
   - 連発防止 state、B 系統メールテンプレート
   - `invest_notify radar run` を追加
5. **Phase 4（A と B の連結）**:
   - A 系統で候補銘柄ヒット時に `radar_hit=true` をメール本文に表示
   - 必要なら A の confirmed/early_warning 上限を絞り、B を主役にする

## 7. ユーザーへの問い（PR でレビューしてほしい点）

1. **A を残すか / B 単独にするか**: 場況把握としての A を残す前提で書いたが、完全に B 単独で良いなら A を削る。
2. **対象市場**: 米株 + 日本株の両方を初手から見るか、米株だけで MVP を作るか。
3. **ファンダ取得の手段**: 無料の Yahoo Finance / Stooq で十分か、有料 API（Polygon, Tiingo, FMP 等）を最初から想定するか。
4. **時価総額バンド**: $500M〜$20B か、$1B〜$30B か。日本株のバンドはどうするか。
5. **通知頻度**: B は週次配信か、芽トリガが出た日だけか。
6. **ユニバース更新**: 半期ごとに手動更新で十分か、自動更新を組むか。
7. **「半信半疑スコア」**: MVP ではカバレッジ件数・出来高・ショート比率の単純合算で構わないか、もう少し凝るか。

これらを決めてもらえれば、Phase 1 の実装に進む。
