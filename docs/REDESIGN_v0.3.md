## 0. 何が問題なのか（現状と目的のズレ）

PR #4, #5 でノイズ削減と確度評価は進んだが、いま毎日届くメールの実態は次のとおり。

- 当日の地政学/政策ヘッドライン
- 当日の 8-K（IR）見出し
- 当日の訴訟/当局発表
- 注視ティッカーの当日ニュース

これは **「日次ニュースモニタ」** であり、**「次の PLTR / APP / SMCI / VRT / CVNA / RKLB / CLS を見つける装置」** ではない。

ユーザーが本当に欲しいのは（要約）：

> **時価総額 10〜100 億ドル程度の、巨大テーマに接続した、まだ市場に半信半疑で見られている会社**
> を、株価が 2〜5 倍になりつつある（あるいは数字が加速し始めた）**胴体の後半** で拾う。

これに対し、現アプリは

- 「**今日のニュース**」を入力にしている → テーマ初期化のシグナルは元々入ってこない
- 「**当日 1 回**」のスナップショット判定 → "売上加速 3 四半期連続" のような **時系列の連続性** が見えない
- 評価軸が **当日の post-return (5〜10営業日)** → "数か月〜2年" の中盤モメンタム評価ができない
- 入力が **RSS 100〜200 件** → 地政学・マクロが大半で、テーマ周辺の中小型株が拾えない

つまり、構造的に「日次ニュースの中から 10 バガー候補が偶然降ってくるのを待つ」装置になっており、毎日の通知に該当銘柄が出てこないのは当然。

---

## 1. 仕切り直しの基本方針

「日次ニュース」と「10 バガー候補」は **別物** として扱う。
ひとつのパイプラインに混ぜず、**2 系統に分離する**。

| 系統                      | 目的                                                 | 周期      | 入口                                                         | 出口                              |
| ----------------------- | -------------------------------------------------- | ------- | ---------------------------------------------------------- | ------------------------------- |
| A. **Daily Watch（既存）** | 場況の把握 / 注視銘柄の動き / 当日の確度高イベント                       | 毎日（月〜土） | RSS（ニュース/IR/訴訟/SNS）                                        | 既存メール（補助的位置づけに格下げ）     |
| B. **Multibagger Radar（新規・主役）** | **テーマ初期〜中盤の米株中小型を継続的に発掘**して、週次で状態を通知 | **週次（月曜のみ）** | FMP（ファンダ）+ ユニバース（時価総額バンド付き米株）+ テーマ辞書 + 価格モメンタム | 週次サマリメール（A とは別件名・別フォーマット）|

つまり、**毎日の通知に 10 バガー候補が出ない** のは仕様であって異常ではない。
**B は「鳴らない週もある装置」** として運用する。

## 4. ユーザーが要求した「狙うべき銘柄」の条件

ユーザーの引用から、機械的に判定可能な条件に落としたもの:

| #  | 条件                              | 取得難度  | 機械判定 | Phase 1 で実装 |
| -- | ------------------------------- | ----- | ---- | ----------- |
| C1 | 時価総額バンド（$500M〜$30B）              | 易     | ◎    | ✅          |
| C2 | 巨大テーマに接続している                    | 中     | △（テーマ辞書 + sector）| ✅（簡易） |
| C3 | 売上成長がすでに数字に出ている（YoY/QoQ）        | 中     | ◎（FMP）| ✅          |
| C4 | 粗利率/営業利益率が改善している（営業レバレッジ）        | 中     | ◎（FMP）| ✅          |
| C5 | バリュエーション過熱（PSR/PER）になっていない      | 中     | ◎（FMP）| ✅          |
| C6 | 増資で株主価値を壊していない（希薄化）             | 中     | ◎（FMP）| ✅          |
| C7 | 直近 3〜4 四半期で「仮説が決算ごとに強くなっている」    | 高     | ○（時系列スコア） | ✅       |
| C8 | 株価モメンタム：底から 2〜5 倍 / 200日線越え     | 易     | ◎    | ✅          |
| C9 | 市場がまだ「地味」「怪しい」「一時的」と見ている         | 高     | △（出来高比 + アナリスト推定数） | ✅（単純合算） |

「悪い中盤」の条件（除外側）:

- 株価だけ先に 5〜10 倍、売上成長は弱い → C5（過熱）で減点
- PSR/PER が異常 → C5 で減点
- 直近決算で大きく外した → C3/C7 で減点

## 5. スコア計算（半信半疑スコアを含む、単純合算）

各シグナルを 0〜1 に正規化し、加重平均する（重みは MVP 段階では均等寄り）。

```
size_band         = 1.0 if $500M <= mcap <= $30B else 0.0
theme_link        = 1.0 if sector in {Tech, Industrials, Energy, Comm} else 0.5
growth            = clamp(latest_qtr_yoy_revenue / 0.30, 0, 1)
margin_improve    = clamp((latest_op_margin - 4q_ago_op_margin) / 0.10, 0, 1)
valuation_room    = 1.0 if PSR < 8 else (1 - clamp((PSR - 8)/8, 0, 1))
non_dilutive      = 1.0 if shares_yoy <= 0.03 else (1 - clamp((shares_yoy - 0.03)/0.10, 0, 1))
consistency_4q    = ratio of 4 quarters where YoY revenue is increasing
momentum          = 1.0 if 200dma reclaimed AND 2x<=底からの倍率<=5x else 0.5 if 200dma reclaimed
skepticism        = 1.0 if (vol20/vol60 <= 1.2) AND (analyst_count <= 8) else 0.5

total = mean([all above])

state = "trigger"     if total >= 0.75 AND momentum >= 0.5
      = "candidate"   if total >= 0.60
      = "watch"       if total >= 0.40
      = "overheated"  if PSR > 16 OR 株価が底から>=10倍
      = "out"         otherwise
```

PSR / アナリスト数 / 出来高は FMP の Starter で取れる範囲で OK。
取れないものは中立（0.5）にして総合スコアを歪めない。

## 7. ユニバース更新（半期手動 + stale 通知）

### 運用フロー

1. **Phase 1**: ユニバースは半期に一度、ユーザーが手動で再生成する
   - `python -m invest_notify radar build-universe --config config.yaml`
   - 出力された `data/radar/universe.json` に `generated_at` が含まれる
2. **古さの判定**:
   - `generated_at` が **180 日**（半年）を超えたら **stale** とする
3. **古い場合の通知**:
   - **A 系統の毎日メール**の冒頭に「⚠ Radar ユニバースが N 日経過しています。`build-universe` を実行してください」と警告セクションを挿入
   - **B 系統の週次メール**にも同セクションを挿入
4. **Phase 2 以降**: 月次自動 + 差分ログを実装し、stale 警告は不要になる

### exclude.yaml / include.yaml

```yaml
# data/radar/exclude.yaml
exclude:
  - ticker: SPAC1
    reason: SPAC 残存
  - ticker: PENNYX
    reason: ペニー化 / 流動性なし

# data/radar/include.yaml
include:
  - ticker: VRT
    reason: ユニバース外でも追跡したい
```

`build-universe` 時にこれらを引いて反映する。

## 9. 実装上の注意

- **FMP Starter は 300 req/分、5年史**: 実 API 検証時点でユニバースは約 **2100 銘柄**（米株 NYSE/NASDAQ、$500M〜$30B、ETF/ファンド除外）。毎週フルスキャンしても十分（初回キャッシュ生成で約 15〜25 分、以降は数分）

### 実 API 検証で観測された現実（2026-05 時点）

実データを通すと以下のような分布になる（要、運用しながら閾値を再調整）:

| 状態 | 件数（フル 2126 銘柄） |
|-----|----|
| trigger（total≥0.75 かつ momentum≥0.5） | **約 540 件** |
| candidate（total≥0.60） | 約 1285 件 |
| overheated（PSR>16 または底から≥10x） | 約 130 件 |

`trigger` が想定より多い（25% が trigger）。原因は size_band / theme_link / valuation_room が 1.0 になりやすく、`growth` と `margin_improvement` の寄与が薄まるため。週次メールで上位 10 件しか詳細表示しないので運用上の支障は小さいが、**件数を絞るなら以下の調整が候補**:

- `trigger` の閾値を total≥0.85 / momentum≥0.7 に引き上げる
- スコア合算を均等から重み付きに変える（growth と margin に強い重み）
- consistency_4q を 0.67 以上に絞り込む

これらは Phase 1.5 の運用調整として、最初の数週分のメールを見ながら決める。
- **キャッシュ必須**: FMP 呼び出しは `data/radar/_fmp_cache/` にローカル保存。ファンダは四半期ごとなので「決算日が更新された時だけ再取得」で十分
- **A と B のメールを混ぜない**: 件名で `[Radar Weekly]` プレフィックスを付ける
- **B の cron 実装方式**: 別 cron / 別 GitHub Actions ワークフローで月曜のみ実行（A の cron は触らない）
- **ロバスト性**: FMP が落ちても A は影響を受けないようプロセス分離
